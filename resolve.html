业务价值：Copilot 不是“回答”，而是“推进处理”

你的 CDN Loader 的效用，要在产品页上体现为：可落地的动作闭环与可度量的业务指标。

输出结构（给人用，也给系统用）

  • 处理策略
    分类、风险分、推荐动作、证据清单、SLA 建议。
  • 客户回复
    多语言模板 + 可替换变量（订单号/物流单号/政策条款）。
  • 动作按钮
    创建工单、升级纠纷、导出复盘摘要（全部可审计）。

为什么必须走 CDN（而不是散落脚本）

  • 稳定发布
    stable/canary 分流，线上故障可快速回滚，不影响业务页代码。
  • 多系统覆盖
    同一份脚本插入订单页/退款页/工单页，能力一致，体验一致。
  • 治理与合规
    把权限、审计、策略集中在 SDK/后端，而不是散落在每个页面。

闭环流程：从“识别退款请求”到“完成处理记录”

这里展示业务闭环的最短路径（可审批、可追踪、可复盘）。

1
识别上下文 从页面信息提取订单/退款原因/物流状态（默认脱敏），作为本次处理的上下文。
2
生成结构化方案 输出分类、风险分、推荐动作、证据清单、SLA；避免“聊天散文”。
3
落到动作 一键生成客户回复 / 创建工单 / 升级纠纷（由人确认，避免越权）。
4
写入记录 把最终选择与证据链写入处理记录，用于审计与复盘看板。

合规安全：企业愿意用的前提

你做的是“业务系统嵌入式 AI”，安全与治理必须在产品页就讲清楚（这也是成交关键）。

默认最小化数据

  • 脱敏优先
    地址/电话/邮箱等 PII 默认不进入提示词；只在必要时以脱敏形式出现。
  • 权限隔离
    按租户、角色授权；不同角色看到的“动作按钮”不同。
  • 审计可追踪
    每次建议/动作都有记录：谁触发、用到哪些证据、最终做了什么决策。

发布治理（这就是 CDN 的价值）

  • 灰度与回滚
    stable/canary 快速切换；线上问题不需要改业务页代码即可回滚。
  • 环境一致性
    同一页面可切 dev/stage/prod；避免“开发可用，上线不可用”。
  • 可观测与成本
    按 project/env/release/tenant 维度度量效果与 token 成本（用于 ROI）。

FAQ

为什么产品页必须把 Copilot “做成动作闭环”，而不是只放一个聊天框？

企业采购看的是“能否推进业务流程”：生成方案、生成回复、创建工单、证据打包、复盘记录与指标闭环。 只聊天无法量化收益，也无法纳入审计与权限体系。

为什么用 CDN Loader 比在每个系统里手写集成更成熟？

CDN 让你把版本、灰度、回滚、安全策略集中治理；业务系统只保留最小挂载点，不被 SDK 细节耦合。

如果我有多个产品线/多个后台系统，怎么复用？

复用同一个 Loader，按 data-project / data-env / data-release 分流；按 data-mount 选择挂载位置； 业务上下文由 SDK 统一采集，减少每个系统二次开发成本。
